## 本次改动
修复球局详情页「确认球局完成」按钮在 **iPhone（iOS）** 上点击无任何反应（连确认框都不弹出）的线上 bug。

## 根因分析
按钮是否可点由前端 `canConfirmComplete = isAuthor && ended && status≠completed && status≠cancelled` 决定。其中 `ended`（球局是否已结束）通过把日期时间拼成字符串再 `new Date()` 解析得到：

```js
// 旧写法（iOS 不兼容）
new Date(('2026-06-28' + ' ' + '20:30').replace(/-/g, '/'))  // => '2026/06/28 20:30'
```

`new Date('2026/06/28 20:30')` 这种「日期 + 空格 + 时分（无秒）」格式在 **iOS 的 JSCore 引擎上会解析为 Invalid Date**（Android / 模拟器 / Node 的 V8 引擎能正常解析，因此一直未被发现）。一旦 `endDt` 为 Invalid Date，`isNaN(endDt.getTime())` 为 true，`ended` 被判为 **false**，`canConfirmComplete` 随之为 false。

此时 wxml 底部按钮栏的 `wx:if/elif` 链不会渲染第 113 行那个带 `bindtap="onConfirmComplete"` 的「确认球局完成」按钮，而是落到「该球局已开始，无法报名」这一**无 bindtap 的灰色禁用按钮**——表现就是「点击毫无反应」。

## 定位证据
- Nginx 访问日志显示用户每次打开球局详情页只发出 `match.getById`，**从未出现 `match.confirmMatchComplete` 请求**，证明 `onConfirmComplete` 根本没被触发（按钮没绑定/没渲染）。
- 数据库确认球局 id=31 status=`full`、结束时间 06-28 20:30 已过、发起人即当前用户，数据层面完全满足确认条件。
- 用户设备为 iPhone（iOS 26.5.1），与 iOS Date 解析兼容性问题吻合。
- 后端 `confirmMatchComplete`（routers.ts 第 2674 行）逻辑无问题，仅校验「存在 + 是发起人 + 未 completed」，status=full 可正常通过。

## 修复方案
新增 iOS 兼容的 `parseDateTime(dateStr, timeStr)` helper：用正则逐字段拆出年/月/日/时/分，再用 `new Date(y, mo-1, d, h, mi)` 构造，全平台解析口径一致。替换 game-detail.js 中三处拼接式 `new Date()`（星期计算、ended、isStarted）。

验证脚本以球局 31 数据模拟（matchDate=2026-06-28, endTime=20:30, now=06-29 12:03）：endDt 正确解析、`ended=true`、`isStarted=true`、week=周日，`canConfirmComplete=true`，按钮恢复可点。
