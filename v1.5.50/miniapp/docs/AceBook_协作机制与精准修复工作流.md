# AceBook AI 团队：协作机制与精准修复 Bug 工作流

> 配套文档：《AceBook 五角色职责定位》《AceBook 架构地图》。
> 本文解决：你下需求/报 bug 后，五个角色如何高效协作，尤其是**如何精准修 bug、不再盲查浪费时间**。

---

## 一、你和团队的协作方式（你只需做三件事）

作为唯一决策人（老板），你在整个流程里只需要：

1. **提出**：一句话说清你要什么，或描述遇到的问题/现象（不需要懂技术）。
2. **拍板**：当我（产品经理角色）给出 A/B/C 方案对比时，你选一个或提调整。
3. **验收**：我交付体验版后，你按我给的"验证清单"扫码测试，确认通过或反馈问题。

其余拆任务、查代码、定根因、写代码、测试、部署、归档，全部由五个角色内部完成，对你透明可见但不需要你操心。

---

## 二、两条主线流程

### 流程 A：你提一个新需求（做新功能 / 改功能）

```
你提需求
  → Leader 澄清意图、定范围、拆阶段
  → 产品经理 写功能规格 + 给方案对比 →【你拍板】
  → 架构师 出技术方案、查架构地图标明改哪里、评估影响面
  → 程序员 按方案改代码 + 真实数据自测
  → Leader 上传体验版、推 GitHub、登记 AGENTS.md →【你验收】
  →（上线后）产品运营 看数据，反馈下一轮
```

### 流程 B：你报一个 Bug（这是重点，解决盲查）

见下方第三节专门的"精准修复 Bug 工作流"。

## 三、精准修复 Bug 工作流（核心，五步法）

> 设计目标：把这次"球局消失"那种"反复试探、查了很多轮"的过程，固化成一套**有顺序、有依据**的流程，靠《架构地图》直接缩小范围。

### 第 0 步：你描述现象（越具体越快）
理想的描述包含三要素：**在哪个页面 + 做了什么操作 + 看到什么异常**。
> 例："约球列表页，我发的 20:00 球局，过了 20:30 就看不到了。"
（有截图更好，能直接看到时间、tab、状态。）

### 第 1 步【架构师】先定位"层"，不要急着翻代码
按《架构地图》的**问题定位速查表**对号入座，先判断问题在哪一层：
- **数据层**？→ 先查数据库，确认数据是否真的丢了/被改了。**"消失/丢失"类问题一律先查库**（铁律1）。
- **接口层**？→ 直接 curl 调线上接口，看返回对不对。
- **前端展示层**？→ 数据和接口都正常，那就是前端过滤/渲染问题。

> 这次"球局消失"：速查表直接指向 booking.js 的 `isHistory`；查库确认数据没丢 → 锁定前端展示层。**省掉所有盲猜。**

### 第 2 步【架构师】定位到"具体文件和函数"
用《架构地图》的"页面↔接口↔表映射"和"关键文件锚点"，定位到具体文件、具体函数、甚至具体行号。
> 例：booking.js 第 277 行 `isHistory`、第 16 行 `computeCountdown`。

### 第 3 步【产品经理 + 你】确认怎么改才合理
如果修复涉及产品取舍（如"球局到点后该不该还显示"），产品经理给 A/B 方案 + 推荐，**你拍板**。
纯技术 bug（如算错、空指针）则跳过此步，直接修。

### 第 4 步【程序员】精准改 + 真实数据验证
- 只改定位到的那几处，不扩大范围（铁律5）。
- **改完必须用真实数据复现验证**（铁律3）——像这次写脚本验证 5 个时刻那样，证明"修好了且没改坏别的"。

### 第 5 步【Leader】上线收口
- 后端改动：重建 dist + pm2 restart（铁律2）。
- 上传新体验版、推 GitHub、登记 AGENTS.md（铁律4）。
- 给你"验证清单"→ **你验收**。

## 四、为什么这套流程能省时间（对比这次的实际经历）

| 环节 | 没有地图（这次的实际过程） | 有地图（以后） |
|---|---|---|
| 定位层 | 反复猜：是取消？是状态？是查询？查了多轮 | 速查表直接指向"前端历史判定" |
| 定位文件 | 逐个翻 booking.js / routers.ts / 数据库 | 锚点直接到 booking.js:277 |
| 确认数据 | 中途才想到查库 | 第 1 步就查库，先排除数据丢失 |
| 结果 | 多轮往返、耗时长 | 几步锁定，快 |

**地图越完整、更新越及时，这种提速越明显。** 这就是为什么架构师的核心 KPI 是"维护好架构地图"。

## 五、知识沉淀机制（让团队越用越聪明）

每次任务结束，必须沉淀，避免下次重复踩坑：

1. **更新《架构地图》**：新增页面/接口/表，或发现新的"常见问题"，补进速查表。
2. **登记 AGENTS.md**：记录本次变更、版本号、踩过的坑（如 OOM、版本错位）。
3. **沉淀"坑"**：凡是"查了很久才发现"的问题，都要写进速查表，让它下次变成"一眼就知道"。

> 例：这次的"settlement-cron 报错无害""ECS 全量 tsc 会 OOM""仓库版本错位"都已沉淀进地图和 AGENTS.md，下次不会再浪费时间。

## 六、角色与产出对照（一页speed dial）

| 你想要 | 找谁（哪个视角） | 给你什么 |
|---|---|---|
| 做个新功能 | Leader → 产品经理 | 功能规格 + 方案选项 |
| 报个 bug | Leader → 架构师 | 根因定位 + 修复 |
| 看看数据/哪里能优化 | 产品运营 | 数据分析 + 迭代建议 |
| 不知道某功能怎么实现的 | 架构师 | 查架构地图给你讲清 |
| 想确认改动安全/能不能回滚 | 架构师 + Leader | 影响评估 + 回滚方案 |

## 七、立即可用

这套机制**现在就能用**。下次你只要说：
- "我要加个 XX 功能" → 我自动走流程 A；
- "XX 页面有个问题：……" → 我自动走流程 B（精准修复五步法）。

你不需要记任何流程，正常提需求/报问题即可，我会在内部按角色协作，并在关键节点把"方案选项"和"验证清单"交给你。
