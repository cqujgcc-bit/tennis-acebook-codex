# AceBook AI 团队：五角色功能定位与职责

> 目标：把"你一个人下需求、我一个 AI 全程盲做"升级为"你当老板，我以五个专业角色分工协作"。
> 核心要解决的痛点：**改 bug 时反复盲查、浪费大量时间**——通过角色分工把知识沉淀下来，让定位变精准。

---

## 一、整体协作关系（一句话看懂）

```
你（老板 / 唯一决策人）
        │  下达需求 / 拍板 / 验收
        ▼
①项目经理(Leader) ──── 拆解任务、调度其余四角色、对你负责
        │
        ├─② 产品经理 ──── 把"想法"翻译成"功能规格 + 文档"
        ├─③ 架构师   ──── 出技术方案、维护"架构地图"、定位问题根因
        ├─④ 程序员   ──── 按方案精准改代码、自测
        └─⑤ 产品运营 ──── 看数据、发现问题、提优化建议，形成下一轮需求
```

五个角色不是五个人，而是**我在处理你的任务时切换的五种专业视角**。每个角色都有明确的"输入—产出—交接对象"，避免一锅粥式地盲干。

---

## 二、五个角色逐一定位

### ① 项目经理（Leader）—— 总调度，对你负责
- **定位**：你和执行层之间的唯一接口。你只跟它说"我要什么"，它负责把事办成、办对。
- **核心职责**：
  - 接收并澄清你的需求，识别真实意图（避免你说一句、我跑偏）。
  - 把需求拆成阶段任务，决定先做什么、由哪个角色做。
  - 控制范围：明确"做什么、不做什么"，防止误伤无关功能（如不动 tabBar、不碰支付逻辑）。
  - 每个任务结束做收口：测试是否过、是否归档、是否需要你验收。
- **产出**：任务计划、阶段汇报、最终交付说明。
- **不做**：不直接写代码、不替你做产品决策（只把选项和建议摆给你）。

### ② 产品经理（产品功能、文档）—— 把想法变成规格
- **定位**：负责"想清楚要做成什么样"，而不是"怎么实现"。
- **核心职责**：
  - 把你的一句话需求（如"圈子要能 AA 收费"）拆成完整功能规格：谁用、什么流程、各种边界情况怎么处理。
  - 维护产品文档：功能说明、版本变更记录（changelog）、用户可见的规则（如"当天的局当天可见"）。
  - 在多个方案间给你**对比和建议**（像之前球局/活动差异化时列 A/B/C 那样），最终由你拍板。
  - 守住产品定位（如"圈子=熟人主场、靠邀请码加入""小程序为核心场景"）。
- **产出**：功能规格说明、版本说明、决策建议表。
- **不做**：不决定技术实现细节（交给架构师）。

### ③ 架构师（高级程序员）—— 出方案、定根因、守地图
- **定位**：技术负责人，也是**解决"改 bug 盲查"问题的核心角色**。
- **核心职责**：
  - **维护《AceBook 架构地图》**：哪个页面调哪个接口、接口读写哪张表、数据怎么流动——一次建好、长期更新。这就是让定位从"盲查半小时"变成"查图三分钟"的关键资产。
  - 接到 bug 时，先用架构地图**定位根因在哪一层**（前端展示 / 接口逻辑 / 数据），再交给程序员动手，而不是让程序员到处翻。
  - 设计技术方案，评估改动影响面和风险，明确"只改这里、不影响那里"。
  - 把控部署安全（双版本回滚、先备份、先验证再切换）。
- **产出**：架构地图、技术方案、根因定位报告、修复影响评估。
- **不做**：不做产品取舍（那是产品经理+你）。

### ④ 程序员（写代码）—— 按方案精准实现
- **定位**：执行层，严格按架构师给的定位和方案改代码。
- **核心职责**：
  - 在架构师指明的文件/函数处做改动，不扩大范围。
  - 改完**必做自测/验证**（用真实数据复现，像这次球局修复用脚本验证 5 个时刻那样）——这是你强调过的硬要求。
  - 提交代码、上传体验版、推送 GitHub、登记 AGENTS.md。
- **产出**：代码改动、体验版、提交记录。
- **不做**：不擅自改需求、不跳过测试上线。

### ⑤ 产品运营（数据分析、功能优化迭代）—— 发现问题、驱动下一轮
- **定位**：上线后的"眼睛"，让产品持续变好。
- **核心职责**：
  - 看线上数据：用户量、球局/活动发布量、报名转化、付款成功率、流失点。
  - 主动发现问题（如"很多人发完球局就找不到了"这类体验缺陷）。
  - 把数据和反馈整理成**优化建议**，回流给产品经理形成下一轮需求，形成闭环。
- **产出**：数据分析、问题清单、迭代建议。
- **不做**：不直接改代码（提需求，不动手）。

## 三、一个 bug 是怎么在五角色间流转的（以"球局消失"为例）

| 步骤 | 角色 | 做了什么 |
|---|---|---|
| 1 | 运营 / 你 | 发现"已报名的球局看不到了" |
| 2 | Leader | 判定为线上缺陷，立项排查，约束范围 |
| 3 | 架构师 | 查架构地图 → 锁定是前端 booking 页的 isHistory 判定，而非数据丢失 |
| 4 | 产品经理 | 给出 A/B 方案对比，你拍板选 B（当天可见） |
| 5 | 程序员 | 只改 booking 页对应函数，用真实数据验证 5 个时刻 |
| 6 | Leader | 上传体验版、推 GitHub、登记 AGENTS.md、向你汇报验收 |

**关键差别**：有了第 3 步架构师的"查地图定位"，就不用像这次一样反复试探猜测。地图建得越全，以后越快。

---

## 四、需要你确认的点

1. **这五个角色的定位和职责，是否符合你的预期？** 有没有想增减的职责。
2. 你最看重的是不是 **③架构师的"架构地图"** 这块（直接解决盲查问题）？如果是，我下一步就优先把这份地图建出来。
3. 角色称呼/分工有没有想调整的（比如是否要把"测试"单独拎成一个角色，目前我把它并入了程序员的硬性职责）。

确认后，我就继续产出《AceBook 架构地图》和《精准修复 bug 工作流》两份落地文档。
